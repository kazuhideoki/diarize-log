# 話者分離文字起こしツール方針

## 目的

各種ツールや利用状況に依存せず、話者分離つきで文字起こしできるようにします。  
文字起こしデータは、Codex CLI などのエージェントで二次成果物を作る前提で扱います。

## 想定する場面

- 対面
- オンラインミーティング

## 全体方針

- 完全リアルタイムは目指さず、ある程度のペースで話者分離つき文字起こしを行います
- 単一プロセスの CLI とします
- 実装言語は Rust、前提環境は macOS とします

## CLI の流れ

- CLI を実行して開始します
- 必要に応じて既知の話者を指定できるようにします
- 終了は Ctrl+C、または一定時間の無音を条件にします
- ただし、開始直後にすぐ終わらないよう最低継続時間を持たせます
- 終了時は、そこまでのデータを文字起こし対象にして通知します

## 音源取得の考え方

- 対面では、まずマイク入力を前提にします
- オンラインでは、自分の声と相手側の音を自然に扱えるようにします
- そのため、system 音声と mic 音声は別録りを基本にします
- 状況に応じて、特定アプリの音声取得も扱える形を目指します

## 文字起こしの考え方

- OpenAI の `gpt-4o-transcribe-diarize` を利用します
- 話者分離つきで文字起こしします
- 既知話者の参照音声と名前を渡せる前提で設計します
- 既知話者の参照は最大 4 人までを前提にします
- モデルはひとまず固定とし、差し替え対応は初期段階では行いません

## データの持ち方

- 1 回の CLI 実行では連続録音を行い、その中から capture 単位(=1リクエスト)の音声を切り出して扱います
- capture ごとの文字起こしデータはまず JSON ファイルで保持します
- 必要になれば capture 管理の sqlite 化を検討します
- マージ後の確定版は 1 つの `final.jsonl` に追記していきます

## 動き方

- セミリアルタイムで文字起こしします
- 数分ごとの capture で更新しつつ、少し重なりを持たせます
- 無音を見て早めに区切ることで、つながりやすくします
- 前後 capture で同じ発話と見なせる場合は、後段で自然につながるようマージします
- capture ごとに発話の切れ方が変わる前提で、置き換えや統合が起きる設計にします
- リアルタイム文字起こしは行いません
